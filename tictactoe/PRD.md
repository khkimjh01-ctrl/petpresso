# 틱택토(Tic-Tac-Toe) 게임 PRD

**문서 버전:** 1.0  
**최종 수정일:** 2025-01-27

---

## 1. 개요

3×3 격자에서 두 플레이어가 번갈아가며 한 칸씩 두고, 가로·세로·대각선 중 한 줄을 먼저 완성하는 쪽이 승리하는 클래식 틱택토 웹 게임의 제품 요구사항 문서입니다.

---

## 2. 게임 규칙

### 2.1 기본 규칙

| 항목 | 설명 |
|------|------|
| **보드** | 3×3 총 9칸의 격자 |
| **플레이어** | 2명 — **X**와 **O** |
| **턴** | X가 선공, 이후 O와 X가 번갈아 한 칸씩 선택 |
| **조작** | 빈 칸을 클릭해 해당 칸에 자신의 마크(X 또는 O)를 둠 |

### 2.2 승리 조건

- **가로·세로·대각선** 중 **한 줄(3칸)**을 자신의 마크로 채운 플레이어가 승리
- 승리하는 줄의 조합(8가지):
  - 가로 3줄: (0,1,2), (3,4,5), (6,7,8)
  - 세로 3줄: (0,3,6), (1,4,7), (2,5,8)
  - 대각선 2줄: (0,4,8), (2,4,6)

### 2.3 무승부

- 9칸이 모두 채워졌는데 승리한 플레이어가 없을 때 **무승부**로 처리

### 2.4 게임 종료 후

- 한 판이 끝나면 **다시 하기**로 보드를 초기화하고 선공(X)부터 새 게임 시작
- 필요 시 **점수**는 유지할지, 초기화할지 선택 가능(기능 정의 시 결정)

## 3. 필요한 기능 목록

### 3.1 필수 기능(Must Have)

| ID | 기능 | 설명 |
|----|------|------|
| F-01 | **보드 렌더링** | 3×3 보드를 화면에 표시하고, 각 칸 클릭 시 해당 플레이어 마크 표시 |
| F-02 | **턴 제어** | 현재 차례 플레이어(X/O) 표시 및 턴에 맞는 칸만 선택 가능하도록 제한 |
| F-03 | **승리 판정** | 가로·세로·대각선 한 줄 완성 시 해당 플레이어 승리 처리 및 게임 종료 |
| F-04 | **무승부 판정** | 9칸 모두 찼는데 승자 없을 때 무승부 처리 및 게임 종료 |
| F-05 | **다시 하기** | 버튼/동작으로 보드 초기화 후 새 게임 시작(선공은 X) |
| F-06 | **승리 줄 강조** | 승리한 3칸을 시각적으로 강조(색상, 테두리, 애니메이션 등) |

### 3.2 권장 기능(Should Have)

| ID | 기능 | 설명 |
|----|------|------|
| F-07 | **점수 기록** | X 승수 / O 승수 / 무승부 횟수 표시 및 유지 |
| F-08 | **반응형 레이아웃** | 모바일·태블릿·데스크톱에서 보드와 버튼이 잘 보이도록 구성 |
| F-09 | **접근성** | 키보드로 칸 선택·확인, 스크린 리더용 텍스트 등 기본 접근성 고려 |

### 3.3 선택 기능(Could Have)

| ID | 기능 | 설명 |
|----|------|------|
| F-10 | **AI 대전** | 1인 모드 — 사용자는 X/O 중 하나, 상대는 AI(O 또는 X) |
| F-11 | **난이도 선택** | AI 난이도: 쉬움(랜덤) / 보통(휴리스틱) / 어려움(미니맥스 등) |
| F-12 | **점수 초기화** | “점수 초기화” 버튼으로 X/O/무 승수 리셋 |
| F-13 | **선공 설정** | 새 게임 시 X/O 중 누가 선공할지 선택 |
| F-14 | **게임 기록(히스토리)** | 최근 N판의 결과를 목록으로 표시 |
| F-15 | **다크/라이트 테마** | 사용자 테마 선택 또는 시스템 설정 따르기 |

## 4. UI/UX 요구사항

### 4.1 화면 구성

- **상단**: 게임 타이틀, 부제(필요 시)
- **정보 영역**: 현재 차례(X 또는 O), 필요 시 점수(X승 / O승 / 무승부)
- **보드**: 3×3 격자 — 각 칸은 버튼 또는 클릭 가능 영역
- **결과 메시지**: “X 승리!” / “O 승리!” / “무승부!” 문구 및 필요 시 간단한 애니메이션
- **조작 영역**: “다시 하기” 버튼, 점수 초기화 등 선택 기능용 버튼

### 4.2 시각적 요구사항

| 항목 | 요구사항 |
|------|----------|
| **보드** | 칸 간 구분이 명확한 그리드, 터치/클릭 영역이 충분히 큼(최소 44px 권장) |
| **X / O** | 두 마크가 색상·모양으로 구분되며, 폰트/아이콘 크기가 읽기 쉬움 |
| **승리 줄** | 배경색, 테두리, 글로우, 라인 등으로 승리한 3칸이 한눈에 구분되도록 표현 |
| **비활성** | 게임 종료 후 칸 추가 선택 불가, 버튼 비활성화 상태로 표시 |

### 4.3 인터랙션

- **칸 선택**: 클릭/터치 한 번으로 즉시 마크 표시, 턴 자동 전환
- **다시 하기**: 클릭 시 보드와 턴만 초기화(점수는 유지할지 기능 스펙에 따라 결정)
- **피드백**: 칸 호버/포커스 시 시각적 피드백, 게임 종료 시 결과 메시지 명확히 표시

### 4.4 반응형

- **모바일**: 보드가 화면 너비에 맞고, 버튼 크기가 터치하기 쉬운 수준
- **데스크톱**: 보드 중앙 정렬, 과도하게 크지 않도록 max-width 등으로 제한 가능

### 4.5 접근성(권장)

- 시맨틱 HTML(`button`, `aria-label` 등) 사용
- 키보드만으로 “다시 하기” 및 칸 선택 가능
- 색상만으로 상태 구분하지 않고, 모양/텍스트/아이콘으로도 구분

## 5. 기술 스택 추천

### 5.1 경량/바닐라 구성(권장 — 학습·배포 간단)

| 구분 | 추천 | 비고 |
|------|------|------|
| **마크업** | HTML5 | 시맨틱 태그, `button` 기반 보드 |
| **스타일** | CSS3 | 변수, Grid/Flex, 미디어쿼리 |
| **로직** | Vanilla JavaScript (ES6+) | DOM 조작, 이벤트 리스너 |
| **개발 도구** | 필요 시 Live Server 등 로컬 서버 | 파일 프로토콜 제한 회피 |

- **장점**: 빌드 없이 브라우저에서 바로 실행, 의존성 최소  
- **적합**: 프로토타입, 포트폴리오, 규모 작은 프로젝트

### 5.2 React 기반(확장·재사용 고려 시)

| 구분 | 추천 | 비고 |
|------|------|------|
| **프레임워크** | React 18+ | 함수형 컴포넌트, Hooks |
| **스타일** | CSS Modules / Tailwind / Styled Components | 팀 선호에 따라 선택 |
| **상태** | `useState` / `useReducer` | 전역 상태 라이브러리 없이 가능 |
| **빌드** | Vite | 빠른 HMR, 설정 단순 |

- **장점**: 컴포넌트 재사용, AI/히스토리 등 기능 추가 시 구조화 용이  
- **적합**: 실무 웹 스택 연습, 이후 기능 확장 계획이 있을 때

### 5.3 Vue 기반(대안)

| 구분 | 추천 | 비고 |
|------|------|------|
| **프레임워크** | Vue 3 (Composition API) | `<script setup>`, `ref`/`reactive` |
| **스타일** | Vue 단일 파일 컴포넌트 내 `<style>` 또는 Tailwind | |
| **빌드** | Vite | Vue 공식 템플릿 사용 가능 |

- **장점**: 문법 단순, 러닝 커브 낮음  
- **적합**: Vue를 쓰는 팀이거나 Vue 학습 목적일 때

### 5.4 기술 선택 가이드

| 상황 | 추천 스택 |
|------|------------|
| 빠른 구현, 의존성 최소 | **HTML + CSS + Vanilla JS** |
| React 실습·포트폴리오 | **React + Vite + CSS/Tailwind** |
| Vue 실습·팀 스택이 Vue | **Vue 3 + Vite** |
| 오프라인/간단 배포 | **Vanilla 또는 React/Vue + 정적 빌드** |
